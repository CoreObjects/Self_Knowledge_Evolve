# 数通域最小实验单元本体重构

## 1. 文档目的

本文在既有网络通信语义知识基础设施方案之上，采用“概念—机制—方法—条件—场景”的五层语义结构，对当前以数通域作为最小实验单元的本体进行重构。

重构目标不是把原有协议分类简单改写，而是把数通域知识从“概念目录”升级为一种可支撑设计、配置、验证与场景化检索的多层本体。

---

## 2. 重构原则

### 2.1 从“协议树”转向“工程知识结构”

旧式本体常按“协议/特性/设备”建树，例如：
- BGP
- OSPF
- MPLS
- ACL
- QoS

这种结构便于收纳定义，但难以表达：
- 为什么某机制在某场景有效；
- 如何组合多个协议能力完成业务；
- 哪些配置对象存在依赖与顺序；
- 何时采用某种方案、何时避免使用。

因此，新本体不再以协议树为唯一主轴，而改为以五层语义结构组织。

### 2.2 本体对象不止概念节点

在数通域中，需要显式建模至少五类知识对象：

- Concept：对象与能力实体
- Mechanism：动态原理与因果机制
- Method：配置、验证、排障、实施方法
- ConditionRule：适用条件、约束、禁忌、决策规则
- ScenarioPattern：典型业务场景与方案模式

### 2.3 以工程闭环作为最小实验单元边界

数通域最小实验单元的边界不是“收完几个协议”，而是至少要支撑下述闭环中的一个子集：

业务需求 → 技术选型 → 逻辑设计 → 配置落地 → 验证验收

因此，本次重构优先选择最具代表性的数通任务簇：

1. 企业园区双出口互联
2. MPLS L3VPN 业务开通
3. EVPN/VXLAN 数据中心二三层互联
4. OSPF + BGP 多域路由互通

- 所有场景要组合概念对象与服务能力。

### 3.2.2 机制层关键关系

### 3.2.3 示例：BGP 不再只是概念节点

在旧本体里，BGP 通常只是一个 `ProtocolObject`。

在新结构里：
- `BGP` 仍然存在于概念层；
- 但与之关联的机制层对象至少包括：
  - `SessionEstablishment`
  - `PathVectorPropagation`
  - `BestPathSelection`
  - `RouteReflection`
  - `CommunityBasedPolicyControl`

这样系统才能回答：
- BGP 是什么；
- 为什么其路由会按某种方式传播；
- 为什么某条路径被选中；
- 为什么 RR 拓扑会影响控制平面收敛。

### 3.3.4 示例：MPLS L3VPN 开通方法

`MPLSL3VPNProvisioningMethod` 可拆分为：

4. 建立 PE-CE 路由关系
5. 确认 P/PE MPLS 转发基础可用
6. 导入导出 VPNv4 路由
7. 验证路由表与业务可达性

这类知识不应作为普通文本片段存在，而应作为方法层对象显式建模。

## 3.4 条件层（Condition Layer）

回答“何时用、何时不用、为什么这样用”。

这是从“知识点”迈向“工程判断”的关键层。

### 3.4.1 条件一级类

#### A. ApplicabilityCondition
- SmallScaleApplicability

- MultiTenantApplicability

- LowLatencyApplicability

#### B. ConstraintCondition
- DeviceCapabilityConstraint
- TeamOAMCapabilityConstraint
- AddressResourceConstraint
- ControlPlaneScaleConstraint
- HardwareFeatureConstraint
- InteroperabilityConstraint

#### C. RiskCondition
- AsymmetricPathRisk
- RoutingLoopRisk
- RouteLeakRisk
- BroadcastDomainExpansionRisk
- FailoverOscillationRisk
- OperationalComplexityRisk

#### D. DecisionRule
- PreferBGPOverStaticForScalabilityRule
- PreferVRFForIsolationRule
- AvoidLargeL2StretchRule

- PreferBFDForFastFailureRule

#### E. ContraindicationRule
- AvoidRedistributionWithoutBoundaryControlRule
- AvoidDefaultRouteOnlyForGranularInterSitePolicyRule
- AvoidUnboundedEVPNFloodingRule
- AvoidVRRPLoadshareUnderCertainStatefulNATConditionsRule

### 3.4.2 条件层表达形式

条件层可采用：

- 规则表达式
- 决策表
- 约束关系
- 风险评分模型
- 适用性区间

示例：

- IF 分支数量持续增长 AND 需统一策略控制 THEN 优先考虑 BGP/路由策略而非大量静态路由。
- IF 业务要求广播域跨站点延伸 AND 规模较大 THEN 警惕二层扩展带来的广播与故障域风险。
- IF 需亚秒级链路故障感知 THEN 引入 BFD 或等效快速检测机制。

### 3.4.3 条件层关键关系

- `applies_to`
- `enables`
- `limits`
- `contraindicates`
- `preferred_under`
- `risky_under`
- `selected_by`

#### C. DataCenterScenario
- EVPNVXLANFabricScenario

#### D. RoutingCollaborationScenario

#### E. ChangeAndValidationScenario
- IncrementalMigrationScenario
- FailoverDrillScenario

### 3.5.2 场景层槽位定义

每个场景模式建议至少包含以下槽位：

- 业务目标
- 网络边界与规模
- 关键约束
- 采用的概念对象
- 采用的关键机制
- 所需方法脚本
- 适用与禁忌条件
- 验证与验收标准
- 风险与回退
- 典型案例参考

### 3.5.3 示例：企业双出口园区场景

`DualExitCampusInternetScenario` 应至少包含：

- 业务目标：互联网访问冗余、核心业务可用性、出口流量控制；
- 关键对象：边界路由器、防火墙、VRRP 网关、默认路由、NAT 策略、QoS 策略；
- 关键机制：FirstHopRedundancy、PrimaryBackupSwitchover、DefaultRoutePropagation、PolicyBasedForwarding；
- 方法：ExitStrategyDesignMethod、DualExitConfigurationMethod、FailoverVerificationMethod；
- 条件：双出口链路能力是否对称、是否允许不对称回流、是否存在状态防火墙；
- 风险：回流、不对称路径、NAT 状态失配、主备抖动；
- 验证：主链路断开验证、路径验证、会话保持验证、业务恢复时间验证。

## 4. 跨层关系设计

五层本体的关键不只是分层，而是跨层连通。

### 4.1 核心跨层关系

- Concept `participates_in` Mechanism
- Mechanism `implemented_by` Method
- Method `governed_by` ConditionRule
- Scenario `composed_of` Concept / Mechanism / Method / Condition
- Scenario `validated_by` VerificationMethod
- ConditionRule `selects` Method / Scenario
- ConditionRule `limits` Mechanism / Concept
- Case `instantiates` ScenarioPattern

### 4.2 典型跨层链路

#### 链路一：从概念到方法
VRF → RouteTargetBasedIsolation → VRFProvisioningMethod

#### 链路二：从机制到条件
BestPathSelection → PreferBGPOverStaticForScalabilityRule

#### 链路三：从场景到配置
DualExitCampusInternetScenario → ExitStrategyDesignMethod → DualExitConfigurationMethod → FailoverVerificationMethod

#### 链路四：从任务问答到知识召回
“某企业双出口园区如何做主备切换？”
→ 场景层召回 `DualExitCampusInternetScenario`
→ 条件层过滤适用模式
→ 方法层返回配置与验证脚本
→ 概念与机制层提供解释与证据

## 5. 与现有语义知识基础设施的对接方式

## 5.1 知识对象类型扩展

原系统中的图数据库节点类型，建议从：
- OntologyClass
- Entity
- Fact
- KnowledgeSegment

扩展为：
- ConceptNode
- MechanismNode
- MethodNode
- ConditionRuleNode
- ScenarioPatternNode
- CaseNode
- EvidenceNode
- KnowledgeSegmentNode

## 5.2 标签体系调整

原有标签体系如果只把 chunk 打到协议节点，会造成“方法/条件/场景信息丢失”。

建议每个知识片段至少支持：

- `concept_tags`
- `mechanism_tags`
- `method_tags`
- `condition_tags`
- `scenario_tags`
- `semantic_role_tags`

这样同一段文本可以同时表达：
- 它讲的是 BGP；
- 它讲的是 BestPathSelection；
- 它是在描述配置方法；
- 它包含某类适用条件；
- 它属于多出口场景。

## 5.3 抽取流水线调整

原流水线中对文本的处理，建议改为：

1. 识别概念对象
2. 识别机制描述
3. 识别方法脚本片段
4. 识别条件句与决策规则
5. 识别场景背景
6. 将片段对齐到五层对象
7. 将多片段融合为规范化对象与规则

这意味着抽取目标从“实体关系抽取”扩展为“多层知识对象抽取”。

## 7. 评测口径变化

在本体重构后，知识评测口径应同步变化。

### 7.1 不再只评估概念覆盖率

还应评估：
- 机制覆盖率
- 方法覆盖率
- 条件规则覆盖率
- 场景覆盖率
- 跨层可连通性

### 7.2 面向任务闭环评测

对于每个核心场景，检查是否可完整支持：

- 场景识别
- 关键概念召回
- 机制解释
- 方法输出
- 条件筛选
- 验证路径输出

### 7.3 面向知识槽位评测

例如 `MPLSL3VPNProvisioningScenario` 的槽位是否完整：
- 业务目标
- 关键对象
- 机制
- 方法
- 条件
- 风险
- 验证

如果场景存在但只填了概念与定义，则说明它仍停留在“概念本体”阶段。

---

## 8. 结论

数通域最小实验单元的本体，不应再被定义为一组协议与设备术语节点，而应被定义为一组可支撑工程闭环的多层知识对象集合。

通过“概念—机制—方法—条件—场景”的五层重构，数通域本体可以从：

- 静态术语目录

升级为：

- 可解释原理、可召回方法、可筛选条件、可组合场景的工程语义骨架。

这将使原有语义知识基础设施真正具备向方案设计、配置指导、验证支撑延伸的结构基础。
